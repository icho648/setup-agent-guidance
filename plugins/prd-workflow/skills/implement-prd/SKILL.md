# 实施 PRD

把已确认的产品契约交付为可验证实现。`AGENTS.md` 约束如何实现；PRD 约束实现什么；ticket 只能收窄本次范围。

## 在 Plan Mode 下

- 只做允许的只读探索，不实施或修改工作区；完成本节后输出可直接交给执行 Agent 的决策完整计划。
- 先固定下文第 1 节的契约与基线，再理解受影响流程、接口、数据流、高风险边界和验证入口。不得仅按 PRD 标题或文件清单猜测方案。
- 输出前读取 [Plan 表达选择](references/plan-expression.md)，识别最难用线性文字理解的关系，并选用最小有效表达。图形、表格、树或线框用于降低理解成本，不是必需装饰；简单改动继续使用短段落或列表。
- 清楚区分 PRD 已确认、从当前仓库推导、实施前仍需核验三类信息。协议、外部服务或易变化事实未经当前权威来源验证时，写成实施检查点，不得伪装为既定方案。
- 计划至少锁定目标与边界、关键实现和接口变化、数据或调用流、高风险失败模式、AC 对应验证及明确假设。视觉表达不能代替必要文字，也不要用另一种格式重复同一信息。

## 1. 固定契约与基线

- 读取工作区规范、PRD；输入为 ticket 时同时读取父 PRD。
- 提取本次目标、范围、非目标、产品不变量、硬约束、AC 和人工决策边界。只实施正式且无阻塞的条目；草稿子范围必须由用户明确确认。
- 确认每项范围内产品行为和硬约束都有正式 AC，并为每个 AC 记录通过条件、验证方式和入口。契约缺失、冲突或无法验证时先请求修订或确认，不得迁就实现改写 PRD。
- 记录实施前 `HEAD`、暂存/未暂存 patch、未跟踪文件及必要文件哈希，明确本次 included / excluded 文件或 hunk。既有改动与本次范围重叠且无法可靠区分时停止并询问。

## 2. 实现并形成可审 Diff

- 先完整理解受影响流程，再实现满足契约的最小完整改动；修复缺陷时定位共享根因和相关调用方，不只修补 ticket 点名路径。
- 优先复用现有实现、标准库、平台原生能力和已安装依赖，不为未确认的未来需求新增抽象、配置、依赖或文件。安全、权限、数据完整性、迁移、状态机、并发、重试等高风险逻辑优先测试驱动；低风险改动按价值复用或补充测试。
- 开发中运行相关测试、类型检查和静态检查，形成范围清晰、可独立审阅的 Diff。只声称实际运行过的检查。
- 数据库、迁移、持久数据或不可逆操作必须先确认环境、数据所有权和恢复方案；默认只在临时或可安全恢复的实例验证。
- 不为制造固定点擅自暂存、提交或覆盖既有改动。

## 3. 执行审阅循环

形成可审 Diff 后、最终完整检查前，加载同插件的 `review-prd-implementation` Skill，传入实施前基线、固定点、PRD/ticket、本次契约范围、当前工作区和已知风险。固定双轴审阅不是可选项。

`review-prd-implementation` 是本 Skill 的硬依赖，必须随插件一同安装。若仅安装了独立的 `implement-prd` 归档而无法加载 `review-prd-implementation`，停止并要求用户补装（安装完整的 `prd-workflow` 即同时获得两者），不得跳过审阅或自行替代审阅逻辑。

主 Agent 只负责评估 findings 的合理性并保持循环：

- `needs_revision`：修改、定向验证并继续原审阅实例；
- `passed`：进入最终验证；
- `needs_decision`：产品取舍、风险接受或授权不足，询问用户；
- `blocked`：客观条件无法继续，说明解除条件。

finding 格式、子 Agent 编排、复用和复审规则只由 `review-prd-implementation` 定义，不在此重复。

## 4. 最终验证与验收

仅在 `review_status=passed` 后运行工作区规定的最终完整检查，再按 [验收报告模板](references/acceptance-report.md) 在聊天中逐条报告。

每条必须包含 AC 编号及具体内容、`Pass / Fail / Not verified`、`Automated / Manual / Observed`、验证入口、直接证据和对应位置。位置使用带绝对路径和单个行号的可点击本地链接；不能跳转 Diff 时不得声称可以。测试全绿不能代替逐条 AC 判断。

默认不另写报告文件、不提交、不推送。验收报告引用审阅流程产生的闭环证据和最终状态，不在此重新推导审阅结论。
