# 审阅 PRD 实现

为同插件的 `implement-prd` Skill 提供最终验证前的审阅闭环。双轴审阅 Agent 只报告问题；主 Agent 独占修改权，并负责判断 findings 是否成立。

## 1. 固定范围

调度审阅前，主 Agent 必须完整读取本次 PRD/ticket；不得只根据文件名、摘要或 Diff 猜测产品契约。建立审阅范围包：实例 ID、实施前基线、固定点、`HEAD`、工作区状态、提交和 Diff 入口、未跟踪文件、included / excluded 文件或 hunk、PRD/ticket 的准确路径及其完整本次契约。

- 仅已提交改动：`git diff <固定点>...HEAD`。
- 含暂存或未暂存改动：`git diff <固定点>`；未跟踪文件另行枚举并直接读取。
- 既有 dirty WIP 默认排除；与本次改动重叠且基线不足以区分时停止并询问。不得为制造固定点擅自暂存或提交。

## 2. 并行首次审阅

使用唯一实例 ID 并行创建两个只读子 Agent，并保存 Agent ID：

- `standards_review_<实例ID>`：依据适用的 `AGENTS.md`、代码与测试规范审查实现；在不改变产品契约的前提下，同时检查可删除代码、重复实现、可由标准库或平台原生能力替代的依赖、单实现抽象和无实际使用方的配置；复杂度 finding 必须给出更小替代方案，硬问题必须引用规则或证明交付风险。
- `spec_review_<实例ID>`：接收 PRD/ticket 的准确路径和完整本次契约，先完整读取原文，再仅依据其范围、非目标、不变量、硬约束和 AC 审查；未完成读取或没有可靠 spec 时不能通过。

所有本轮已启动的首次报告全部返回前，主 Agent 只负责固定范围、调度和收集结果，不得因某一份报告先返回就提前核验、修订或运行检查，也不自行通读 Diff 寻找缺陷，不运行以核验实现正确性为目的的测试、lint 或代码检查，不形成第三份独立审阅报告。此时只允许确认基线可区分、范围可读取及必要 Agent / 工具可用。

两者都接收完整范围包。每项 finding 包含稳定编号（`STD-*` / `SPEC-*`）、严重度、文件与紧凑行范围、证据、影响和建议；没有问题写 `No findings`。严重度：`P0/P1` 阻塞，`P2` 默认不阻塞（标为 blocker 必须证明不可交付），`P3` 为建议。单纯减少行数不构成 blocker；复杂度 finding 除非违反适用规范或造成可证明的交付风险，否则只能是 `P2/P3`。分别输出 `## Standards` 和 `## Spec`，不得跨轴合并。

## 3. 判断、修订、复审

取得 Standards、Spec 的全部首次报告后，主 Agent 再统一综合评审，逐项核验范围、引用、代码证据、可复现性和最坏影响，记录：

- `已修复`：改动位置与理由；
- `不适用`：推翻 finding 的证据；
- `非阻塞建议`：仅用于不影响交付的 P2/P3，记录不纳入本次的理由和剩余影响；
- `需要产品决策`：停止并询问；
- `已接受风险`：仅在用户确认具体最坏影响后使用。

结论明确且不改变产品契约时自主修改，运行受影响的定向检查，并记录二次改动范围。涉及产品取舍、规则冲突、安全或数据风险、不可逆变化、授权不足或判断信心不足时返回 `needs_decision`，不要替用户选择。

复审时通过客户端可用且能触发新一轮执行的 follow-up、resume 或等价续跑机制复用受影响的原 Standards / Spec Agent，发送原 findings、主 Agent 处置、二次改动、检查结果和当前完整范围包；要求按 `已解决 / 仍存在 / 新发现` 输出，保留旧编号。原 Agent 不可用时创建替代 Agent，并通过上述材料恢复上下文。修订触及共享接口、安全、并发、迁移或产品语义时重扫完整 Diff。

只要经主 Agent 核验成立的问题仍可在当前契约内修复，就继续同一实例，不设机械轮次上限。同一 blocker 连续两轮无新证据、修复振荡或无法可靠判断时，转为人工决策；缺少 spec、无法区分改动或必要双轴 Agent/工具不可用时才 `blocked`。

## 4. 状态与证据

主 Agent 每轮产出：

- `needs_revision`：仍有可自主修复的 blocker；
- `passed`：所有已取得且成立的 findings 均已处理，无未接受 blocker；
- `needs_decision`：需要产品决定、风险接受或授权；
- `blocked`：缺少 spec、范围不可分或必要能力不可用。

证据至少包含范围包、两个 Agent 及首次报告、每项处置、二次改动与检查、复审结果、未解决 blocker 和待决策项。

## 5. 提交审阅报告

结束本 Skill 前，主 Agent 必须在聊天中按上一节“证据至少包含”的字段和顺序提交完整审阅报告；这些字段与 `implement-prd` 验收报告的“审阅闭环”部分一致，独立调用时也不减少。不得只返回状态或压缩摘要，也不要在本 Skill 重复定义另一套报告字段。

只有 `passed` 且上述要求的完整审阅报告已提交，才返回 `implement-prd` 的最终验证阶段。默认不另写报告文件，不暂存、不提交 Git、不推送。
