# PRD 参考模板

优先使用工作区模板，并映射以下必需语义。没有答案时写“待定”并说明是否阻塞，不得猜测。

# <产品或功能名称>

- 状态：草稿 / 可交付实施
- 版本或基线：<如工作区有约定则填写>
- 最后实质更新：<本地日期>
- 本次变化：<初始基线或实质变更摘要>

## 核心摘要

只压缩后文，不新增产品内容。

- 问题：<谁在什么情境下遇到什么问题>
- 目标：<希望产生什么变化>
- 边界：<本次提供什么、不提供什么>
- 完成标志：<最关键的可验证结果>

## 1. 问题、用户与依据（必需）

- 目标用户与情境：
- 当前问题：
- 事实与来源：
- **假设 1**：<如何验证>
- **已确认决策 1**：<决定、确认记录、生效基线；用户沉默不算确认>
- **待定问题 1**：<由谁决定；影响；是否阻塞>

### 建议（不属于产品基线）

- **建议 1**：<推荐方案及依据；确认后写入对应正式条目>

## 2. 目标结果与成功标准（必需）

- **目标 1**：<期望发生的用户或业务变化；来源>
  - 成功信号：<可观察、可衡量或可明确判断>

没有可信基线时，不编造数值目标。

## 3. 范围与预期产品行为（必需）

每条行为同时定义需求和范围，不另写重复的场景或功能清单。

- **产品行为 1**：当 <情境> 时，产品应 <可观察行为或结果>。（对应：目标 1；来源）
- **产品行为 2**：当 <异常或边界情境> 时，产品应 <失败、降级或恢复行为>。（对应：目标 1；来源）

### 非目标

- **非目标 1**：<本次明确不覆盖的用户、情境或能力；来源>

## 4. 产品不变量与硬约束（必需）

- **产品不变量 1**：<实现演进时必须保持的产品行为或数据语义；来源>
- **硬约束 1**：<安全、隐私、合规、业务、协议、兼容或迁移边界；来源>

通用工程规则只引用适用的工作区文件。

## 5. 验收规则（必需）

正式 AC 只能引用已确认的产品行为、不变量或约束。

- [ ] **AC-01**：在 <前提> 下，当 <行为> 时，应 <可观察结果>。（对应：产品行为 1）
- [ ] **AC-02**：当 <异常或边界条件> 时，应 <明确结果>。（对应：产品行为 2、硬约束 1）
- [ ] **AC-03**：交付结果保持 <产品不变量>。（对应：产品不变量 1）

每条产品行为、产品不变量和硬约束至少被一个 AC 覆盖。建议项需要验证时，在建议下描述验证思路，不创建正式 AC。

## 6. 演进与自治边界（必需）

### 实施 Agent 可自主决定

- 实现方式、内部结构和常规技术选择，只要满足产品契约、工作区规范和现有测试。
- <其他明确授权的空间>

### 必须由人确认

- 改变目标、非目标、产品行为、不变量、约束或验收基线。
- 引入新的外部承诺、高代价依赖、不可逆迁移或明显的体验变化。
- <项目特有边界>

- 确认记录：
- 新基线：

不得为了迁就当前实现而静默降低验收标准。
