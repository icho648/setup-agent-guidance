# 代码审查与测试标准

本文件是生成项目级 `code_review.md` 的骨架。生成时用仓库证据和用户确认内容替换占位符，并删除所有生成说明；不要把本模板原样安装为空泛规范。

## 审查目标与优先级

按以下顺序寻找能够实际影响用户、生产系统或维护成本的问题，并根据项目特点调整顺序：

1. 正确性与需求符合度。
2. 安全、隐私和数据完整性。
3. 回归风险与兼容性。
4. 可靠性、并发、资源和错误处理。
5. 性能与可运维性。
6. 可维护性；只报告会造成实际风险的风格问题。

项目特别关注：`<经用户确认的风险领域>`。

## 必查项目

### 行为与正确性

- 变更实现了请求的行为，没有遗漏关键分支和边界条件。
- 错误路径、空值、超时、重试和部分失败得到合理处理。
- 新行为与现有调用方和公开契约一致。

### 测试与验证

- 新增或更新的测试覆盖了有意义的成功路径和失败路径。
- 测试能够在修改前暴露问题，或明确证明新增行为。
- 不通过削弱断言、跳过测试或过度 mock 来制造通过结果。
- 完成结论附有最新、可复现的验证证据。

### 安全与数据

- 输入验证、授权、敏感信息、注入风险和依赖边界得到检查。
- 数据库、持久化格式或迁移具有兼容、回滚和失败恢复方案。

### 兼容性与运维

- `<支持的平台、版本和公开接口兼容要求>`。
- 日志、指标和错误信息足以定位新增失败模式，且不会泄露敏感信息。

### 可维护性

- 实现保持必要的最小复杂度，不引入无当前用途的抽象或依赖。
- 名称、边界和注释能解释非显而易见的决定，而不是复述代码。

## 发现严重级别

- **阻塞**：可能造成安全事件、数据损坏、重大功能错误或无法安全发布。
- **重要**：明确的缺陷、回归或高概率维护风险，应在合并前修复。
- **建议**：低风险但有具体收益的改进，不应伪装成阻塞问题。

每条发现都应指出具体位置、触发条件、实际影响和最小修复方向。没有可操作发现时明确说明，并列出尚未验证的剩余风险。

## 完成标准

- 所有阻塞门禁通过，并报告运行命令与结果。
- 实际行为与需求、项目指南及兼容性约束一致。
