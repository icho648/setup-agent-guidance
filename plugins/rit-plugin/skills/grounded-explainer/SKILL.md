# 落地解释器

从具体场景和既有问题出发，帮助用户理解对象为何存在、真正独特的地方、必要实现、实际影响与边界。不要把功能列表、仓库导览或研究过程当作解释。

## 工作流

1. 确认当前请求包含 `$grounded-explainer`、`/rit-plugin:grounded-explainer` 或 `/grounded-explainer`；否则不启用本技能。
2. 完整读取并遵循 `references/explanation-workflow.md`。其中的“权威回答顺序”是唯一推理顺序，不把它机械转换成固定章节。
3. 只检查足以验证核心与边界的源码、配置、日志或一手文档；无法验证对象内部实现时明确说明，不用宿主包装层或架构猜测代替。
4. 根据用户目标和明确的长度要求决定解释深度，以及使用自然语言、简短对比、简单文本流程、Mermaid、伪代码或精确代码；不要机械添加任何形式。
5. 发送前按 reference 中的检查项精简重复和无关内容，但保留必要因果链与用户明确要求。

## 核心约束

- 区分同类对象的普通能力与当前语境中的独特核心，并明确说出差异及影响。
- 实现细节只有在改变理解、使用或评价时才展开；展开时停留在对象内部边界。
- 先给出最小但完整的因果链，再按问题需要展开；用户明确要求一句话或简短回答时优先尊重。
- 用真实场景和机制承载解释；机制讲清后，可加入一个能映射回真实参与者且确实降低理解成本的短比喻。
- 后续修改采用增量方式，保留用户未明确否定的既有解释层次。
