# 交互式课程规范

只有在交互比对话或真实操作更有效时，才创建 HTML。目标是缩短反馈循环，
不是建设课程网站。

## 选择载体

优先级如下：

1. 真实环境中的操作、解释或诊断；
2. 对话中的回忆、追问和反馈；
3. Markdown 检查表或参考资料；
4. 需要重复练习、自动反馈、分支、模拟或可视化时才使用 HTML。

适合 HTML 的例子：调用链排序、日志诊断、架构取舍题、语法练习、带参数的
模拟器和间隔复习。不适合的例子：只问一次的问题、可以直接运行的真实任务，
以及只为了更好看而包装的说明文。

## 课程契约

每个 `lessons/NNNN-<slug>.html` 只包含一个可观察成果，并在开头说明：

- 完成后能做什么；
- 前置能力；
- 什么表现算通过；
- 预计用时。

推荐顺序：

1. **先回忆或预测**：提交前不展示答案。
2. **最少讲解**：只补当前任务需要的知识，并链接一手资料。
3. **针对性练习**：围绕常见误解出少量高信号题。
4. **即时反馈**：说明为什么对、为什么错，以及下一次如何判断。
5. **迁移任务**：改变条件，或要求在真实环境中完成一步。
6. **结果摘要**：允许复制回答、分数、尝试次数和待复习项。

课程末尾提醒学习者可以向智能体追问，但不要把“看完页面”写成完成条件。

## 出题规则

根据要验证的能力选择题型：

- **自由回忆**：适合术语、流程、心智模型；先输入，再显示核对标准。
- **预测题**：先预测输出、状态或故障位置，再运行或揭示结果。
- **选择题**：适合区分相近概念和常见误解，不适合证明开放式设计能力。
- **排序/配对**：适合流程、调用链、优先级和映射。
- **情境诊断**：给出日志、症状或约束，让学习者选择下一项检查。
- **迁移题**：改变一个关键条件，要求重新判断或解释取舍。

选择题应满足：

- 只有一个最佳答案，除非明确写“多选”；
- 错误选项来自真实误解，而不是明显笑话；
- 选项长度、语法、具体程度和格式相近；
- 正确答案的位置不要形成固定规律；
- 提交后解释判断依据，不只显示“正确/错误”。

无需强求每个选项字符数完全相同；这种机械限制可能让表达失真。目标是消除
提示线索，而不是制造别扭文本。

开放题不得使用关键词匹配冒充理解判断。使用“显示核对标准”，或让学习者把
答案复制回对话获得反馈。

## 反馈与证据

即时反馈至少包含：

- 当前回答是否符合核对标准；
- 关键判断依据；
- 如果错误，最小修正提示；
- 可以重试的入口。

题目成绩是诊断信号，不是掌握结论：

- 单次选择题通常最多支持“识别”；
- 能解释答案和排除错误选项，可支持更强的过程证据；
- 能处理变化情境或真实任务，才支持“独立完成”或“迁移运用”。

浏览器中的本地状态不自动写入 `MASTERY.md`。只有用户明确提交结果、复制
摘要，或智能体直接观察了答题过程时，才作为证据记录，并注明来源。

## HTML 与组件要求

首次需要交互课程时：

1. 创建 `.learning/lessons/` 和 `.learning/assets/`；
2. 从技能目录 `assets/interactive-lesson/` 复制 `template.html`、
   `lesson.css` 和 `quiz.js`；
3. 把模板保存为下一个 `NNNN-<slug>.html`；
4. 替换示例内容，复用共享 CSS/JavaScript；
5. 检查 HTML、JavaScript 语法，并在浏览器中完成一次键盘操作和答题验证。

实现要求：

- 默认离线可用，不依赖 CDN、远程字体或分析脚本；
- 使用语义化 HTML、`fieldset`/`legend`、明确的 `label` 和可见焦点；
- 即时反馈使用 `aria-live`，键盘可完成全部操作；
- 支持窄屏、系统明暗主题、`prefers-reduced-motion` 和打印；
- 不收集隐私数据，不自动上传答案；
- 本地持久化默认关闭；确需使用时说明保存位置和清除方式；
- 不把正确答案当作安全秘密，客户端测验只用于学习。

视觉设计服务于阅读和反馈。保持清晰、克制、一致，不为单节课程引入框架。
