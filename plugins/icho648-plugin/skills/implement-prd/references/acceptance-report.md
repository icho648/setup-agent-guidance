# 验收报告模板

只记录支持交付决策的信息。

## 实施摘要

- PRD 或 ticket：
- PRD 状态与基线：
- Git 固定点与既有改动：
- 本次契约范围：<目标、产品行为、非目标、产品不变量、硬约束、AC>
- 排除的建议、待定和非目标：

## 验收结果

| AC | 具体验收内容 | 结果 | 方式与入口 | 直接证据 | 对应位置（如有） |
|---|---|---|---|---|---|
| AC-01 | 写出该 AC 的完整、具体要求，不得只写编号或模糊摘要 | Pass / Fail / Not verified | Automated / Manual / Observed；测试、页面、API、数据库或 CLI | 命令、测试名、输出摘要、截图或未验证原因 | `[实现](/绝对/工作区/路径/file.py:123)`、`[测试](/绝对/工作区/路径/test_file.py:45)`；没有单一位置时明确说明 |

位置规则：

- 优先链接直接实现与直接验证该 AC 的文件行号，避免堆砌无关位置。
- 使用可点击的本地 Markdown 链接：绝对路径加单个行号。
- 链接只用于打开工作区文件对应行；不要声称能跳转到审阅 Diff hunk。
- 不使用 `::code-comment` 作为验收报告导航或验收证据链接。
- 人工或外部系统验收没有代码位置时，链接 PRD 对应 AC；仍无位置时写“无单一文件位置”。

- `Pass`：直接证据符合 PRD。
- `Fail`：结果不符合 PRD。
- `Not verified`：尚无足够证据；说明风险。

## 审阅闭环

- 首次审阅：审阅实例 ID、固定点、实施前基线、included / excluded Diff 范围、Standards / Spec Agent 名称与轮次
- Standards findings：编号、严重度、是否 blocker、结论、主 Agent 处置与证据
- Spec findings：编号、严重度、是否 blocker、结论、主 Agent 处置与证据
- 二次改动范围：文件、hunk 或行范围
- 复审：是否复用原 Agent；逐项已解决、仍存在和新发现
- 降级或未接受风险：Agent 不可用、上下文恢复方式、待用户决策事项
