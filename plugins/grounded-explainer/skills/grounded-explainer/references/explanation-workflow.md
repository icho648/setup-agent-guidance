# 落地解释工作流

本文件是详细解释流程的唯一来源。先确定当前问题中的核心，再选择证据、深度与表达形式。

## 权威回答顺序

解释工具、库、产品组件或代码库时，按以下顺序保留所有相关步骤；仅当某一步确实不适用或不会改变理解时省略：

1. 把对象放进具体、真实的场景，说明谁在做什么、哪里出现问题。
2. 在对象出现前，说明直接方案或同类对象通常能做什么，以及它们在当前压力下为何不足。
3. 引入对象，明确它在当前语境中的独特核心，以及这个差异为何重要。
4. 明确判断实现是否值得展开，并说明理由。
5. 值得展开时，解释产生独特行为的内部机制；否则说明常规胶水为何不影响当前理解。
6. 回到开头的场景，说明机制带来的可观察结果。
7. 说明主要边界或取舍。
8. 源码检查影响答案时，在末尾给出决定性的实现引用。

流程图、伪代码、精确代码、额外示例或更深章节只能补充这条主线，不能替代其中的场景、问题、核心判断、实际影响或边界。

## 判断真正核心

核心不是自动等于最底层实现、调用链、生命周期或最复杂功能，而是：去掉后，对象在当前问题中便不再成立的最小观念、设计选择或行为。

写作前在内部回答：

1. 没有这个对象时，什么问题仍然存在？
2. 它主要用哪个设计动作或行为解决问题？
3. 这会产生什么可观察结果？
4. 它有意把什么责任留给其他层？
5. 用户需要理解核心机制，还是知道结果已经足够？

将候选细节分为三类：

- **必要信息：** 缺少后会形成错误心智模型，必须解释。
- **支撑信息：** 用于连接问题、设计、机制与结果。
- **偶然信息：** 不改变答案的创建步骤、供应商清单或周边架构，省略或只作为末尾引用。

研究决定可信度，用户意图决定披露范围。不要因为检查时发现了某个细节，就自动写进答案。

## 区分普通能力与独特核心

不要把完整功能列表当作核心。对比最直接的做法或同类对象的常规基线：

- 这类对象通常本来就具备什么？
- 当前场景中的哪种压力让常规方案变得重复、不安全、别扭或不足？
- 这个对象集中、隔离、保证或替换了什么？
- 这个差异会怎样影响用户或周边系统？

将结论限定为“当前语境中的独特核心”，除非有比较证据，不夸大成全市场绝对独有。明确写出类似判断：“普通能力是 A；这里真正值得关注的是 B，因为 C。”

## 判断实现深度

满足以下至少一项时展开实现：

- 独特核心依赖不直观的算法、抽象、协议、对象模型或控制边界。
- 实现决定重要行为、限制、扩展点、性能特征或取舍。
- 用户询问如何工作、为何这样设计、要求结合源码，或需要实现指导。

实现只是常规胶水、只能证明功能存在，或不会改变理解、使用和评价时，不展开。介绍工具或库时应简短说明这个判断，而不是静默跳过。

## 检查证据与对象边界

准确性依赖实际情况时，只检查足以验证核心和边界的源码、配置、日志或当前一手文档。仅当创建、交接、调用者或数据流本身就是核心时继续追踪。

解释代码库内部实现前：

1. 找到对象自身的仓库、子模块、内嵌副本、已安装包或当前官方源码。
2. 检查公共入口和产生独特行为的内部注册、策略、适配、算法或转换过程。
3. 仅在返回抽象或转换边界能补全机制时继续检查。
4. 宿主调用位置只用于确定输入与输出背景。

宿主应用选择配置并调用对象，只能证明集成存在，不能解释对象如何实现核心。无法检查对象自身源码时，说明内部机制尚未验证。

## 选择表达形式

自然语言是默认方式。线性内部路径使用简单文本流程；只有存在有意义的内部阶段、分支、策略交互或状态变化时才使用 Mermaid。选择或分发规则决定结果时可加入伪代码；语法、API 契约、类型或错误路径本身重要时才使用精确代码。

图和伪代码必须来自已检查的对象源码，并停留在输入进入对象到输出离开对象的边界内：

- 流程图展示内部阶段与状态变化。
- 伪代码展示内部选择、分发、转换、回退或不变量。
- 同时使用时，两者必须提供不同信息。
- 把对象画成单个方框只是集成图，不是内部实现图。

代码片段必须最小且有决定性，优先取自真实调用或实现位置；省略与核心无关的导入、初始化和样板错误处理。

## 写作边界

- 使用真实参与者、输入、动作、请求、文件、记录或错误建立场景；默认不使用比喻。
- 不强行制造戏剧化问题。对象若只是减少重复或统一接口，就直接这样说。
- 长度由完整因果链决定，不设默认句数、段落数或代码量上限；完整后再删除重复、清单堆砌和无关分支。
- 正文使用职责层级名称。除非精确符号或 API 契约本身就是解释对象，否则把类名、函数名和文件路径放到末尾实现引用。
- 不输出功能清单、架构导览、阅读列表、历史或研究日记。
- 用户提出后续修改时，以增量方式更新；除非用户明确否定或要求替换，不删除此前已接受的解释层次。

不同问题仍使用同一权威顺序，只调整重点：概念强调情境与机制，比较强调决定性差异，故障强调证据与根因，文档结合源码强调文档段落到文件的映射，面试回答强调约束、选择与取舍。

## 发送前检查

- 事实、内部机制、流程图、伪代码与引用是否来自已检查来源，而非推断冒充验证？
- 每个实现细节是否都在解释独特核心，而非周边胶水？
- 表达形式是否确实比自然语言更清楚，并且没有重复同一信息？
- 正文是否把原始仓库符号翻译成职责名称，只保留决定性引用？
- 是否只删除重复与无关内容，没有删除用户要求或必要因果步骤？
- 是否在回答边界处结束，没有漂移到相邻主题？
