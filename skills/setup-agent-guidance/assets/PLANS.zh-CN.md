# Codex 执行计划（ExecPlan）

本文档描述执行计划（“ExecPlan”）的要求。ExecPlan 是一份设计文档，编码代理可以依照它交付一个可工作的功能或系统变更。假定读者对本仓库完全陌生：他们只有当前工作树和你提供的这一份 ExecPlan，不记得以前的计划，也没有任何外部上下文。

## 如何使用 ExecPlan 和 PLANS.md

编写可执行规格（ExecPlan）时，必须严格遵循 `PLANS.md`。如果它不在当前上下文中，应完整读取 `PLANS.md` 以刷新记忆。彻底阅读并在必要时重读源材料，从而形成准确的规格。创建规格时，从骨架开始，在研究过程中逐步补充完整。

实施 ExecPlan 时，不要询问用户“下一步做什么”；直接推进到下一个里程碑。始终保持所有章节为最新状态，在每个停止点新增或拆分进度条目，明确记录已经完成的工作和后续步骤。自主解决歧义，并频繁提交代码。

讨论 ExecPlan 时，把决定记录到计划中的日志里，以便长期追溯；任何规格变更的原因都应清晰无歧义。ExecPlan 是活文档，并且始终应该能够仅凭 ExecPlan 本身重新开始工作，而不依赖其他上下文。

研究具有困难要求或重大未知因素的设计时，使用里程碑实现概念验证、玩具实现或其他能够验证用户方案是否可行的原型。通过查找或获取依赖库的源代码进行深入研究，并用原型为完整实现提供依据。

## 要求

不可妥协的要求：

* 每一份 ExecPlan 都必须完全自包含。自包含意味着计划当前版本包含让新手成功实施所需的全部知识和指令。
* 每一份 ExecPlan 都是活文档。随着进展、发现和设计决定的形成，贡献者必须修订它；每次修订后仍须保持完全自包含。
* 每一份 ExecPlan 都必须让完全不了解本仓库的人能够端到端实现该功能。
* 每一份 ExecPlan 都必须产生可以实际演示的工作行为，而不只是为了满足定义而修改代码。
* 每一份 ExecPlan 都必须用日常语言解释所有专业术语；无法解释时就不要使用。

先说明目的和意图。开头用几句话从用户视角解释这项工作为何重要：变更后用户能够做什么以前做不到的事情，以及如何观察它确实工作。然后引导读者完成实现该结果的确切步骤，包括修改什么、运行什么以及应该观察到什么。

执行计划的代理可以列出和读取文件、搜索内容、运行项目及测试。它不知道任何先前上下文，也不能从更早的里程碑推断你的意图。重复写明所依赖的假设。不要把读者指向外部博客或文档；如果某项知识不可或缺，就用自己的话将其写入计划。如果 ExecPlan 基于先前的 ExecPlan，且该文件已经纳入仓库，可以通过引用它来纳入上下文；否则，必须包含旧计划中的全部相关上下文。

## 格式

格式和外层结构应保持简单而严格。每一份 ExecPlan 都必须放在一个标记为 `md` 的单一代码围栏中，以三个反引号开始和结束。不要在其中嵌套其他三反引号代码围栏；需要展示命令、终端记录、差异或代码时，应在这一代码围栏内使用缩进块。使用缩进保证清晰，避免意外提前结束 ExecPlan 的代码围栏。每个标题后留两个换行，使用 `#`、`##` 等正确的 Markdown 标题和列表语法。

当 ExecPlan 写入一个内容仅包含该计划的 Markdown 文件时，省略外层三反引号。

使用朴素的叙述性文字。优先使用完整句子而不是列表。除非简短表达会损害清晰度，否则避免检查清单、表格和很长的枚举。检查清单只允许出现在 `进度` 章节，并且该章节必须使用检查清单。叙述性章节应以段落为主。

## 指导原则

自包含和朴素语言至关重要。如果引入并非日常用语的词语，例如“守护进程”“中间件”“RPC 网关”或“过滤器图”，应立即给出定义，并提醒读者它在本仓库中如何体现，例如指出相关文件或命令。不要写“如前所述”或“按照架构文档”。即使有所重复，也应把需要的解释写在当前计划中。

避免常见失败模式。不要依赖未定义的术语。不要把功能描述得过于字面和狭窄，以至于最终代码虽然可以编译，却没有产生有意义的行为。不要把关键决定推给读者。当存在歧义时，在计划中解决它并解释选择该方案的理由。宁可详细解释用户可见的效果，也不要过度规定无关紧要的实现细节。

用可观察的结果锚定计划。说明用户在实现后可以做什么、需要运行哪些命令以及应该看到什么输出。验收标准应表达为人类可验证的行为，例如“启动服务器后，访问 [http://localhost:8080/health](http://localhost:8080/health) 会返回 HTTP 200 和正文 `OK`”，而不是“添加了 `HealthCheck` 结构体”这样的内部属性。如果变更完全位于内部，也要解释如何演示其影响，例如运行一个修改前失败、修改后通过的测试，并展示使用新行为的场景。

明确说明仓库上下文。使用完整的仓库相对路径命名文件，准确命名函数和模块，并说明新文件应该创建在哪里。涉及多个区域时，先用一小段文字解释各部分如何配合，让新手能够有信心地导航。运行命令时，写明工作目录和确切命令。结果依赖环境时，说明假设，并在合理情况下给出替代方案。

保证幂等和安全。编写能够重复执行而不会造成损坏或漂移的步骤。如果某个步骤可能中途失败，说明如何重试或调整。如果必须执行迁移或破坏性操作，明确说明备份或安全回退方法。优先采用可逐步验证、附加式的修改。

验证不是可选项。写明如何运行测试，适用时如何启动系统，并观察它完成有用的行为。为新增功能或能力说明全面测试方法。写出预期输出和错误信息，让新手能够判断成功或失败。在可能的情况下，展示如何证明变更不仅仅能够编译，而是真正生效，例如通过一个小型端到端场景、CLI 调用或 HTTP 请求与响应记录。给出适合项目工具链的确切测试命令，并解释如何理解结果。

保存证据。当步骤产生终端输出、短小差异或日志时，将其作为缩进示例放在单一代码围栏内。保持简洁，并只保留能够证明成功的内容。需要包含补丁时，优先使用限定在单个文件的差异或读者可以通过计划步骤重建的小片段，不要粘贴大段内容。

## 里程碑

里程碑应当是叙事，而不是官僚手续。将工作拆成里程碑时，先用一小段文字说明范围、里程碑结束时将新增什么、需要运行的命令以及预期观察到的验收结果。像讲故事一样保持可读：目标、工作、结果、证据。进度和里程碑彼此不同：里程碑讲述过程，进度追踪细粒度工作；两者都必须存在。不要只为了简短而缩写里程碑，也不要遗漏可能对未来实现至关重要的细节。

每个里程碑都必须能够独立验证，并以增量方式实现执行计划的总体目标。

## 活计划与设计决定

* ExecPlan 是活文档。形成关键设计决定时，更新计划，同时记录决定本身和背后的思考。把所有决定写入 `决定日志` 章节。
* ExecPlan 必须包含并维护 `进度`、`意外与发现`、`决定日志` 和 `结果与回顾` 章节。这些章节不可省略。
* 发现优化器行为、性能取舍、意外缺陷、逆操作或取消应用语义等影响方案的情况时，在 `意外与发现` 中记录观察结果和简短证据，优先使用测试输出。
* 实施过程中改变方向时，在 `决定日志` 中说明原因，并在 `进度` 中反映影响。计划既是自己的检查清单，也是下一位贡献者的指南。
* 完成重大任务或整份计划时，在 `结果与回顾` 中总结已经实现的内容、仍未完成的事项和得到的经验。

# 原型里程碑与并行实现

可以并且通常应该明确加入原型里程碑，以降低大型变更的风险。例如，在依赖库中添加一个低层操作以验证可行性，或者在测量优化器效果的同时探索两种组合顺序。保持原型具有附加性和可测试性。清楚标明“原型”范围，说明如何运行和观察结果，并写出采纳或放弃原型的标准。

优先先做保持测试通过的附加式代码修改，再执行删除。迁移期间保留适配器和旧路径等并行实现，只要能降低风险或让测试持续通过，就是合理的。说明如何验证两条路径，以及如何在测试保护下安全移除其中一条。涉及多个新库或功能区域时，考虑创建彼此独立的探索性实现，分别证明外部库符合预期并能独立提供所需功能。

## 优秀 ExecPlan 的骨架

    # <简短、以行动为导向的描述>

    本 ExecPlan 是活文档。实施过程中必须始终保持 `进度`、`意外与发现`、`决定日志` 和 `结果与回顾` 章节为最新状态。

    如果仓库中纳入了 PLANS.md，请在此处写明它从仓库根目录起算的路径，并注明本计划必须依照 PLANS.md 维护。

    ## 目的与全貌

    用几句话说明此次变更完成后用户将获得什么，以及如何观察它确实工作。写明将启用的用户可见行为。

    ## 进度

    使用复选框列表总结细粒度步骤。每个停止点都必须记录在这里；必要时，把部分完成的任务拆分为“已完成”和“剩余”两部分。本章节必须始终反映工作的真实现状。

    - [x]（2025-10-01 13:00Z）已完成步骤示例。
    - [ ] 未完成步骤示例。
    - [ ] 部分完成步骤示例（已完成：X；剩余：Y）。

    使用时间戳衡量进度速度。

    ## 意外与发现

    记录实施期间发现的意外行为、缺陷、优化或洞见，并提供简洁证据。

    - 观察：……
      证据：……

    ## 决定日志

    使用以下格式记录工作期间作出的每项决定：

    - 决定：……
      理由：……
      日期/作者：……

    ## 结果与回顾

    在重要里程碑或完成时总结结果、差距和经验，并与最初目的进行比较。

    ## 上下文与定位

    假定读者一无所知，描述与任务相关的当前状态。使用完整路径命名关键文件和模块。定义所有非显而易见的术语。不要引用先前计划。

    ## 工作计划

    用叙述性文字说明编辑和新增内容的顺序。对于每项修改，指出文件和位置（函数、模块），并说明插入或修改什么。保持具体而精简。

    ## 具体步骤

    写出要运行的确切命令和工作目录。命令产生输出时，提供一小段预期记录供读者对照。实施过程中必须持续更新本章节。

    ## 验证与验收

    说明如何启动或操作系统以及应该观察什么。以具体输入和输出表达行为验收。如果涉及测试，应写成“运行 <项目测试命令>，预期 <N> 项通过；新测试 <名称> 在修改前失败、修改后通过”。

    ## 幂等性与恢复

    如果步骤可以安全重复执行，应明确说明。如果某一步存在风险，提供安全重试或回滚路径。完成后保持环境整洁。

    ## 产物与说明

    包含最重要的终端记录、差异或代码片段，使用缩进示例展示。保持简洁，并只保留能够证明成功的内容。

    ## 接口与依赖

    给出明确要求。写明要使用的库、模块和服务及其原因。规定里程碑结束时必须存在的类型、trait/接口和函数签名。优先使用稳定名称和路径，例如 `crate::module::function` 或 `package.submodule.Interface`。例如：

    在 `crates/foo/planner.rs` 中定义：

        pub trait Planner {
            fn plan(&self, observed: &Observed) -> Vec<Action>;
        }

遵循上述指导后，一个无状态代理或完全不了解项目的人都应该能够从头到尾阅读 ExecPlan，并产生一个可工作、可观察的结果。这就是标准：自包含、自给自足、能够指导新手，并且以结果为中心。

修订计划时，必须确保修改完整反映在所有章节中，包括活文档章节；同时在计划底部写一条说明，解释修改内容及原因。ExecPlan 对几乎所有事项都不只描述“做什么”，还要描述“为什么”。
