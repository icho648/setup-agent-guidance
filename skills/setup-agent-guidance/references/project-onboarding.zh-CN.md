# 交互式项目建档流程

## 目标

用可验证的仓库事实生成精简项目指南和代码审查标准。不要机械读取每个文件，也不要把常识、模型能力或未经确认的个人偏好写成项目规则。

## 只读扫描

优先检查：

1. 项目根目录、已有 `AGENTS.md`、`CLAUDE.md`、README、贡献指南和架构文档。
2. 包管理器 manifest、锁文件、构建配置和主要入口。
3. CI 工作流、PR 模板、CODEOWNERS 和发布配置。
4. 测试目录及测试、lint、format、typecheck、覆盖率和安全工具配置。
5. 数据库迁移、公开 API、协议、持久化格式和部署入口。
6. 少量代表性业务模块及其测试，用于理解实际约定是否与配置一致。

跳过依赖目录、构建产物、缓存、二进制、大型生成文件、凭据和密钥。除非用户要求，不运行会改写文件的 formatter、迁移、代码生成或安装命令。

## 形成候选项目画像

根据证据整理：

- 关键目录、模块边界和运行入口；
- 可以从配置确认的启动、构建、测试和静态检查命令；
- CI 中实际阻塞合并的门禁；
- 公开接口、数据迁移、安全或性能高风险区域；
- 已有工程约定和完成标准；
- 证据冲突、失效命令和仍不确定的事项。

每项结论都应能指向具体仓库文件。不要根据语言或框架习惯编造项目规则。

## 与用户确认

先展示精简发现，再询问只有用户能决定的事项。每轮最多提出三个短问题，优先确认：

1. 哪些命令或检查必须阻塞合并；
2. Review 的风险优先级，例如正确性、安全、兼容、性能或 UI；
3. 仓库无法证明的团队偏好，例如 TDD、覆盖率、支持版本、迁移和严重级别政策。

如果用户不回答，保留未确认项，不自行升级为强制规则。不要询问能够从仓库配置、CI 或文档发现的问题。

## 起草与确认

1. 使用 `assets/AGENTS.template.md` 的项目区块格式生成 `agent-guidance:project` 区块，删除不存在的命令和空标题。
2. 使用 `assets/CODE_REVIEW.template.md` 生成根目录 `code_review.md`，删除全部占位符和生成说明。
3. 保持内容短、具体、可执行。测试命令必须写明工作目录和用途。
4. 展示将写入的内容或与现有文件的差异，并等待用户明确确认。
5. 获得确认后才写入。若用户只批准其中一部分，只应用获批部分。

## 更新已有规则

- `agent-guidance:core` 可以按模板自动更新。
- `agent-guidance:project` 和 `code_review.md` 属于用户拥有的项目规则；重新扫描后只提出差异，不静默覆盖。
- 现有内容与仓库事实冲突时，明确指出冲突及证据，让用户决定保留、修改或删除。
- 不删除 managed blocks 之外的内容。
